Lexware-Rechnungsmails wickeln den Nachrichtentext in verschachtelte zentrierte Blöcke (erkennbar am internen templateBody-Marker) und verwenden dort teils Web-Fonts (z.B.     Info
    CASTLE ACME ohne Azure Communication Services
       — warum Graph API + Exchange ausreicht
    
  

  

    

      Früher war geplant, Azure Communication Services (ACS) zu nutzen, um CASTLE
      ACME-Antwortmails an den CA-Server zu senden — weil Exchange Online die Mails so stark
      verändert (ARC-Seal, DKIM, Thread-Topic, CTE 7bit → quoted-printable, ~25 interne Header),
      dass CASTLE die Antwort als invalid ablehnt, und auf Azure-VMs Port 25
      outbound gesperrt ist.
    

    

      Gelöst in v1.1.0: Das Gateway sendet die ACME-Antwortmail über
      Graph API sendMail — Exchange routet sie über den Outbound-Connector zurück
      zum Gateway. Der Handler erkennt Subject: Re: ACME: und ruft
      _rebuild_acme_reply() auf: extrahiert nur den reinen ACME-Response-Block,
      baut ein frisches MIME mit CRLF-Zeilenenden auf und leitet es sauber weiter.
      So erreicht CASTLE eine RFC 8823-konforme Mail — ohne ACS, ohne Port 25.
    

    

      Voraussetzungen:
      RemoteDomain Castle ACME für castle.cloud mit
      ContentType=MimeText, TNEFEnabled=$false, ByteEncoderTypeFor7BitCharsets=Use7Bit
      verhindert zuverlässig die CTE-Konvertierung (Schritt A im Header Observatory oben).
      Der Double-Hop (Graph → Exchange → Connector → Gateway → Castle) ist by design.
