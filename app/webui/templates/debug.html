ACME ACME HTTP-Proxy — nur für CASTLE-Verbindung — optional, leer = Direktverbindung

Optionaler Proxy nur für die ACME/CASTLE-HTTP-Aufrufe (new-order, finalize, etc.) — Graph-API, EXO-SMTP und alles andere bleiben unverändert. In der Regel nicht nötig: Der frühere CASTLE-500-FileNotFoundError bei Erneuerungen lag NICHT an der Absender-IP, sondern an einem CASTLE-Bug bei bestehendem Zertifikat — behoben seit v1.4.392 (Widerruf vor Neuausstellung). Diese Option bleibt als Werkzeug für andere Netzwerk-/CA-Szenarien erhalten.

ACME ACME Account Key zurücksetzen — erzwingt neue Account-Registrierung bei CASTLE

Löscht den gemeinsamen ACME-Account-Key und die Account-URLs (Staging + Production). Beim nächsten Erneuerungsversuch wird automatisch ein neuer Key und ein neuer Account bei CASTLE registriert. Bestehende S/MIME-Zertifikate sind nicht betroffen.

Lab Exchange Header Observatory — testet welche Header Exchange bei ACME-Antwortmails hinzufügt

Schickt eine Test-ACME-Reply via Graph API sendMail durch Exchange. Das Gateway fängt die Nachricht auf dem Outbound-Connector ab und speichert den exakten Raw-MIME — so sieht man welche Header Exchange hinzufügt bzw. verändert.
Optional: RemoteDomain für castle.cloud setzen, dann erneut testen und die Ergebnisse vergleichen.

Schritt A — RemoteDomain für castle.cloud

Setzt ByteEncoderTypeFor7BitCharsets=Use7Bit, ContentType=MimeText, TNEFEnabled=$false — verhindert eventuell die CTE-Konvertierung 7bit → quoted-printable. Keine Auswirkung auf andere Empfänger-Domains.
